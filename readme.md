# 人工智能课程大作业项目说明

## 一、项目名称

基于 Bottom-Up-Agent 的自底向上智能体学习与演示系统

---

## 二、项目简介

本项目是人工智能课程的大作业，参考开源项目 **Bottom-Up-Agent** 的设计思想，围绕“智能体如何在缺乏人工先验知识的情况下，通过观察环境、执行动作、积累经验并逐步形成技能”这一问题，完成一份可展示、可分析、可说明的课程实践项目。

与传统依赖人工预设流程的智能体不同，Bottom-Up-Agent 强调“自底向上”的能力形成方式。智能体从屏幕图像、鼠标点击、键盘输入等基础交互出发，在探索过程中逐步发现有效操作，并将其沉淀为可重复调用的技能。该项目 README 中将这一过程概括为 Zero-prior Learning、Unified Codebase、Experience-Driven Evolution 和 Visualization，并展示了其在 Slay the Spire、Civilization V 等环境中的应用。

---

## 三、项目背景与意义

近年来，大语言模型和多模态模型的发展推动了智能体系统的快速进步。许多现有 Agent 更偏向“自顶向下”的任务执行模式，即开发者先设计好目标、流程和工具调用方式，智能体再按既定流程完成任务。这类方法虽然在结构化任务中效果较好，但也存在对人工规则依赖较强、泛化能力有限、面对复杂开放环境时适应性不足等问题。

Bottom-Up-Agent 提出了一种不同的思路：让智能体不依赖预定义目标、API 或游戏知识，而是从原始像素输入与模拟鼠标、键盘动作出发，通过经验不断发现、改进和共享技能。对于人工智能课程而言，这种设计非常适合作为“大作业”主题，因为它能综合体现视觉感知、语言模型推理、动作执行、技能学习和系统集成等多个 AI 核心问题。

---

## 四、项目目标

本项目的主要目标如下：

1. 学习并理解 Bottom-Up-Agent 的整体设计思路；
2. 梳理其“环境感知—动作执行—技能生成—技能演化”的系统流程；
3. 完成对开源项目核心模块和运行方式的分析；
4. 结合人工智能课程要求，形成一份结构完整、逻辑清晰、可用于展示汇报的课程大作业文档；
5. 展示自底向上智能体与传统流程型智能体的差异及特点。

---

## 五、项目的主要内容

本项目主要围绕 Bottom-Up-Agent 的框架展开，内容包括：

### 1. 智能体总体思想分析

分析其“从低级动作到高级技能”的学习机制，理解智能体为什么不需要在一开始就拥有完整的任务流程。

### 2. 系统模块分析

从代码结构入手，分析项目中的主控模块、视觉模块、动作模块、记忆模块、搜索模块与可视化模块之间的关系。

### 3. 运行流程梳理

结合配置文件与主程序入口，梳理项目从读取配置、初始化智能体、与环境交互、执行动作到保存日志和可视化展示的全过程。仓库中的 `run.py` 负责读取 YAML 配置文件并实例化 `BottomUpAgent`，随后以预设任务启动运行；`config/` 目录中则提供了 `game.yaml` 面向不同环境与模型组合的配置文件。

### 4. 课程化总结

将开源项目的前沿研究思想整理为适合课堂展示和课程答辩的内容，包括项目背景、技术路线、模块作用、分工安排和实验体会等。

## 六、项目特色

1. **强调自主学习** 
   智能体不是依靠固定流程工作，而是在环境中逐步学习有效技能。

2. **结合视觉、动作与语言推理** 
   本项目同时涉及图像感知、鼠标键盘执行以及大语言模型推理，不是单一模型调用，而是多模块协同完成任务。

3. **具有可视化与可分析性** 
   项目支持通过可视化方式展示技能树、调用日志和执行过程，便于分析智能体的行为路径，也更适合课程展示与答辩说明。

4. **适合人工智能课程大作业** 
   该项目兼具理论性、系统性和展示性，能较好体现人工智能课程中的综合实践能力。

---

## 七、A组工作

A组对应“游戏交互执行与可视化平台”，核心职责不是做策略推理，而是把系统真正接到《杀戮尖塔》运行环境上，完成“看到画面—接收动作—执行操作—反馈结果—展示过程”的执行闭环。其输入主要是游戏窗口画面、B组返回的动作决策信息以及用户给定的开始、暂停、单步执行等控制指令；其输出主要是对游戏的实际鼠标键盘操作、执行结果反馈、运行日志与可视化界面信息。

结合子项目方案，A组工作可以进一步概括为以下几个方面：

1. **游戏环境接入**：负责游戏窗口捕获、当前画面维护、基础区域定位与场景接入稳定性处理；
2. **动作执行与流程控制**：负责将 B组 输出的结构化动作转化为点击卡牌、选择目标、选择路径、结束回合等具体操作；
3. **状态展示与可视化**：负责运行控制页面、状态展示区、日志面板、结果可视化与演示界面优化；
4. **反馈与日志记录**：负责记录点击位置、动作类型、执行耗时、成功/失败次数、异常情况和关键截图，并将执行前后结果反馈给 B组。

若进一步细分到组内成员，则可理解为：A1 更偏向场景接入与前端展示，A2 更偏向动作执行、接口联调、异常处理与日志回传。

---

## 八、B组工作

B组对应“状态感知与策略学习引擎”，核心职责是理解《杀戮尖塔》当前局势，并给出下一步动作建议，同时把有效操作沉淀为可复用经验。其输入主要是 A组 传来的游戏截图、界面元素信息、任务目标、历史执行日志以及已有技能/经验记录；其输出主要是结构化动作指令、动作解释、状态判断结果、技能评估结果和经验库更新信息。

结合子项目方案，B组工作可以进一步概括为以下几个方面：

1. **状态识别与建模**：对战斗、地图、奖励、事件等不同场景进行状态抽象，识别手牌、能量、血量、敌方意图和路线选择等关键信息；
2. **策略决策与动作生成**：根据当前局势生成下一步最优动作建议，并输出结构化动作指令与动作解释；
3. **经验沉淀与技能复用**：记录不同状态下动作的成功率、失败率、重复次数和复用次数，持续维护经验库与技能库；
4. **联调与效果评估**：根据 A组 返回的执行结果对决策效果进行评估，并持续优化策略准确率和异常处理能力。

若进一步细分到组内成员，则可理解为：B1 更偏向状态识别与数据建模，B2 更偏向策略选择、经验检索、技能更新与效果评估。

## 九、代码的作用（按 A组/B组任务重新划分）

如果仅按照文件名逐个解释代码功能，虽然可以说明“每个模块做了什么”，但不容易体现其与 A组、B组任务分工之间的关系。为了更适合课程汇报与答辩展示，本文将代码划分为**公共主控模块、A组主责模块、B组主责模块以及 A/B 协同接口模块**四部分。这样既能体现工程结构，也能更清楚地说明系统是如何运行起来的。

### 9.1 公共主控与运行支撑模块

这部分代码不单独属于 A组 或 B组，而是整个系统启动、配置和统一调度的公共底座。

#### 1. `run.py`

项目入口文件，负责读取配置文件、创建 `BottomUpAgent` 实例，并启动智能体运行流程。它相当于系统总启动器，决定整个项目从哪里开始运行。

#### 2. `config/`

用于存放不同环境、不同模型和不同探索模式下的配置文件 `game.yaml`。这部分属于系统的“参数层”，用于统一管理环境类型、分辨率、模型接口和运行模式。

#### 3. `base_model/`

包含 `gpt.py`、`claude.py`、`UI_TARS.py` 等文件，主要负责封装底层大模型接口，为系统提供推理、对话和决策能力支持。它本质上是 B组 决策能力的重要支撑层，但在工程结构上属于整个系统的公共基础模块。

#### 4. `BottomUpAgent/common/BottomUpAgent.py`

这是智能体主体控制类所在文件，负责把感知、决策、执行、记忆和日志等模块组织起来，形成完整运行主线。它可以理解为系统的“总控中枢”：既要调 A组 负责的画面获取和动作执行，也要调 B组 负责的状态理解和策略生成。

---

### 9.2 A组主责代码：交互执行与可视化平台

A组的核心任务是“接游戏、做执行、给反馈、做展示”，因此凡是与画面接入、元素定位、鼠标键盘执行、流程控制、运行展示有关的代码，都可以划入 A组 主责模块。

#### 1. `BottomUpAgent/group_a/Eye.py`

主要负责屏幕图像获取与基础视觉输入处理。对应 A组 方案中的“游戏窗口捕获”和“当前画面维护”任务，是整个执行平台的输入端。没有它，系统连游戏当前的画面状态都无法获取，后续分析和操作都会失去依据。

#### 2. `BottomUpAgent/group_a/Detector.py`

主要负责界面元素检测、区域识别和目标定位。对应 A组 方案中的“基础区域定位”任务，用于帮助系统判断卡牌区域、按钮区域、地图节点区域等可交互位置。它也是 A组 把“看见屏幕”进一步变成“知道该点哪里”的关键桥梁。

#### 3. `BottomUpAgent/group_a/Hand.py`

主要负责鼠标、键盘等具体交互操作。对应 A组 方案中的“动作调度与执行”任务，是把 B组 输出的结构化动作真正落到游戏界面上的执行器，例如点击卡牌、点击怪物、选择路线、结束回合等。

#### 4. `BottomUpAgent/group_a/UnifiedOperation.py`

主要负责对不同环境下的操作动作进行统一封装。对应 A组 方案中的“动作接口规范化”和“流程控制”任务。它的作用类似一个操作翻译层：把不同模块给出的动作描述整理成统一格式，便于执行模块直接调用，也便于日志记录和后续扩展到其他环境。

#### 5. `BottomUpAgent/group_a/visualizer.py`

主要负责结果展示、过程可视化、技能树或日志调用链展示。对应 A组 方案中的“状态展示区、日志面板、结果可视化和整体演示效果优化”任务，是系统对外展示最直观的一层。

#### 6. 与 A组 密切相关的辅助职责

虽然不一定单独体现在某一个文件名上，但 A组 还承担执行日志、关键截图保存、异常提示、运行状态展示、单步/暂停/重置等控制逻辑。这些内容通常会散布在 `BottomUpAgent/common/BottomUpAgent.py`、执行模块和可视化模块之中，共同构成“交互执行与可视化平台”。

---

### 9.3 B组主责代码：状态感知与策略学习引擎

B组的核心任务是“理解局势、生成动作、积累经验、复用技能”，因此凡是与状态抽象、策略选择、经验记忆、搜索规划和动作解释有关的代码，都可以划入 B组 主责模块。

#### 1. `BottomUpAgent/group_b/Brain.py`

主要负责推理、规划和策略选择，可理解为智能体的核心决策模块。它直接对应 B组 方案中的“状态判断”“动作生成”“动作解释”任务，是系统从“看到了什么”走向“下一步该做什么”的关键一跳。

#### 2. `BottomUpAgent/group_b/LongMemory.py`

主要负责长时记忆管理，用于保存经验、历史状态、技能记录和可复用知识。它对应 B组 方案中的“经验库维护”和“技能复用”任务，使智能体不只是当场拍脑袋，而是能把有效策略沉淀下来，后续反复调用。

#### 3. `BottomUpAgent/group_b/Mcts.py`

对应 Monte Carlo Tree Search（蒙特卡洛树搜索）模块，主要用于候选行为搜索与动作选择。它对应 B组 方案中的“策略决策与动作优化”任务，适合在多个可选操作之间评估收益，帮助系统从若干备选动作中找到更优选择。

#### 4. `BottomUpAgent/group_b/Teacher.py`

可理解为对智能体进行引导、评价或反思辅助的模块。它对应 B组 方案中的“效果评估”“反思修正”“经验更新”等任务，帮助系统在执行完成后总结这次动作是否有效、是否应纳入经验库。

#### 5. `base_model/` 与 B组 的关系

虽然 `base_model/` 在工程上属于公共基础层，但从任务角度看，它主要服务于 B组 的状态理解、推理决策与动作解释能力。因此在汇报时可以把它说明为“由公共模块提供、主要被 B组 智能决策引擎调用的底层模型支撑”。

### 9.4 A组与 B组 的接口闭环关系

1. A组 通过 `BottomUpAgent/group_a/Eye.py` 和 `BottomUpAgent/group_a/Detector.py` 获取当前游戏画面与基础元素信息；
2. 这些状态信息被送入 `BottomUpAgent/group_b/Brain.py`、`BottomUpAgent/group_b/Mcts.py`、`BottomUpAgent/group_b/LongMemory.py` 等 B组 模块，用于状态理解、动作决策和经验检索；
3. B组 输出结构化动作指令和动作解释；
4. A组 再通过 `BottomUpAgent/group_a/UnifiedOperation.py` 与 `BottomUpAgent/group_a/Hand.py` 将动作转化为实际点击、拖拽或键盘操作；
5. 执行结果、界面变化、日志与截图被重新反馈给 `BottomUpAgent/common/BottomUpAgent.py` 与 B组 模块；
6. 最终由 `BottomUpAgent/group_a/visualizer.py` 展示当前状态、动作过程、日志结果和技能演化信息。

因此，从课程汇报角度看，这套代码并不是零散文件的堆砌，而是围绕“感知—决策—执行—反馈—经验更新”形成的协同系统。其中 A组 更偏执行平台与展示平台，B组 更偏智能分析与经验学习，`run.py`、`config/`、`BottomUpAgent/common/BottomUpAgent.py` 等公共模块则负责把两组工作真正串起来。

### 9.5 展示数据的结构

为了让 A组 的交互执行平台与 B组 的状态感知和策略决策模块能够稳定联调，项目中需要定义统一的数据结构。这样做的目的，是把“看到什么、决定做什么、执行后发生了什么、经验是否更新”全部标准化，避免不同模块之间各说各话。

从整体流程看，系统中的核心数据主要包括：**环境状态数据、动作决策数据、执行反馈数据、运行日志数据、经验/技能数据** 五类。

#### 1. 环境状态数据（A组传给 B组）

该部分数据由 A组 负责组织，主要来源于游戏窗口截图、界面检测结果和当前运行上下文，用于让 B组 判断当前局势。

可展示为如下结构：

```json
{
 "scene_type": "battle",
 "floor": 12,
 "hp": 65,
 "max_hp": 80,
 "energy": 3,
 "hand_cards": [
  {"name": "Strike", "cost": 1, "type": "attack"},
  {"name": "Defend", "cost": 1, "type": "skill"}
 ],
 "enemies": [
  {"name": "Jaw Worm", "hp": 38, "intent": "attack"}
 ],
 "map_options": [],
 "reward_options": [],
 "screen_image": "current_frame.png",
 "detected_regions": {
  "hand_area": [120, 600, 980, 760],
  "end_turn_button": [1080, 520, 1240, 620]
 }
}
```

其中：

- `scene_type` 表示当前场景类型，例如战斗、地图、奖励、事件；
- `floor` 表示当前层数；
- `hp`、`energy` 等字段描述角色当前状态；
- `hand_cards` 表示当前手牌信息；
- `enemies` 表示敌方信息与意图；
- `screen_image` 和 `detected_regions` 主要由 A组 提供，作为 B组 识别与决策的基础输入。

#### 2. 动作决策数据（B组传给 A组）

该部分数据由 B组 输出，表示“下一步要执行什么动作”。A组 接收到该结构后，将其翻译为具体鼠标或键盘操作。

可展示为如下结构：

```json
{
 "action_type": "click_card",
 "target": {
  "card_name": "Strike",
  "enemy_index": 0
 },
 "reason": "当前能量足够，优先使用攻击牌压低敌方血量",
 "confidence": 0.87,
 "source": "Brain+Mcts",
 "timestamp": "2026-03-10 14:30:21"
}
```

其中：

- `action_type` 表示动作类型，例如点击卡牌、点击敌人、选择路线、结束回合；
- `target` 表示动作目标；
- `reason` 表示动作解释，方便日志展示和答辩说明；
- `confidence` 表示模型或策略对该动作的置信程度；
- `source` 表示本次动作主要来自哪个策略模块。

#### 3. 执行反馈数据（A组回传给 B组）

动作执行完成后，A组 需要将结果反馈给 B组，用于判断动作是否成功，以及是否需要调整经验库。

可展示为如下结构：

```json
{
 "action_type": "click_card",
 "execute_status": "success",
 "before_scene": "battle",
 "after_scene": "battle",
 "time_cost_ms": 426,
 "click_position": [512, 688],
 "screen_diff": "enemy_hp_changed",
 "error_message": "",
 "screenshot_after": "after_action_001.png"
}
```

其中：

- `execute_status` 表示动作执行是否成功；
- `before_scene` 与 `after_scene` 用于判断执行前后状态是否发生预期变化；
- `time_cost_ms` 用于统计动作耗时；
- `click_position` 记录实际点击位置；
- `screen_diff` 用于描述执行后的主要变化；
- 若失败，则 `error_message` 中可记录异常原因。

#### 4. 运行日志数据（A组记录并展示）

运行日志主要用于可视化展示、调试分析和答辩材料整理。

可展示为如下结构：

```json
{
 "step_id": 15,
 "scene_type": "battle",
 "input_summary": "检测到2张可用攻击牌，敌方准备攻击",
 "decision": "使用 Strike 攻击 0 号敌人",
 "result": "success",
 "time_cost_ms": 426,
 "screenshot": "step15.png",
 "operator": "A组执行平台"
}
```

#### 5. 经验/技能数据（B组维护）

该部分数据用于记录可复用经验、技能触发条件和动作效果，是 B组 的核心沉淀内容。

可展示为如下结构：

```json
{
 "skill_id": "battle_attack_single_001",
 "scene_type": "battle",
 "trigger_condition": "单个敌人血量较低且我方能量>=1",
 "action_pattern": "优先打出低费攻击牌",
 "success_count": 18,
 "fail_count": 4,
 "reuse_count": 11,
 "last_update": "2026-03-10 14:45:32"
}
```

通过上述五类数据，整个系统就形成了清晰的数据闭环：

**A组提供状态输入 → B组生成动作决策 → A组执行动作并返回反馈 → B组更新经验 → A组展示结果与日志。**

#### 2. 各文件夹下应放什么代码

##### （1）项目根目录 `project_root/`

放置项目入口和全局配置文件。

- `run.py`：主程序入口，负责读取配置并启动系统；
- `.env`：保存 API Key、环境变量等敏感配置；
- `README.md`、`requirements.txt`、`environment.yml`等。

##### （2）`config/`

放置 YAML 配置文件。

这一层主要保存不同实验环境、不同模型和不同参数设置：

- `game.yaml`

这些文件不写核心逻辑，只负责管理“系统如何运行”的参数，例如模型类型、环境名称、日志路径、分辨率设置等。

##### （3）`base_model/`

放置底层模型接口封装代码。

例如：

- `gpt.py`：OpenAI 模型接口；
- `claude.py`：Claude 模型接口；
- `UI_TARS.py`：其他可选模型接口。

这一层的作用是统一大模型调用方式，让上层决策模块不必关心不同模型 API 的细节。

##### （4）`BottomUpAgent/common/`

放置系统公共主控代码。

- `BottomUpAgent.py`：系统总控，负责协调 A组 与 B组 各模块；
- 若后续还有统一调度器、公共状态机、主循环封装等，也可放在该目录。

##### （5）`BottomUpAgent/group_a/`

放置 A组 主责模块代码。

包括：

- `Eye.py`：画面获取与视觉输入；
- `Detector.py`：界面元素检测与区域定位；
- `Hand.py`：鼠标键盘动作执行；
- `UnifiedOperation.py`：统一动作封装与执行接口；
- `visualizer.py`：日志、过程、技能树或运行结果展示。

##### （6）`BottomUpAgent/group_b/`

放置 B组 主责模块代码。

包括：

- `Brain.py`：状态理解与动作决策；
- `LongMemory.py`：经验库、技能库与长时记忆管理；
- `Mcts.py`：动作搜索与候选策略选择；
- `Teacher.py`：策略评估、反思与经验更新辅助。

##### （7）`weights/`

放置模型权重文件，不放业务代码。

例如 README 中提到的 SAM 权重文件就应放在该目录下。这类文件通常体积较大，只负责支撑视觉检测等能力，不参与具体流程控制。

##### （8）`logs/`

放置运行日志，不放核心算法代码。

可进一步分为：

- `run_logs/`：整体运行日志；
- `action_logs/`：动作级别日志；
- 后续也可继续增加 `error_logs/` 记录异常。

##### （9）`data/`

放置结构化数据文件，便于演示与后续分析。

推荐分为：

- `states/`：A组 传给 B组 的状态数据；
- `actions/`：B组 输出的动作决策数据；
- `feedback/`：A组 执行后的反馈数据；
- `skills/`：B组 维护的经验/技能数据。

##### （10）`screenshots/`

放置截图结果。

推荐分为：

- `current/`：当前调试截图；
- `history/`：历史关键截图、答辩展示截图。

#### 3. A组与 B组 代码放置的对应关系

为了和分工文档保持一致，也可以进一步概括为：

- **A组负责的代码主要放在**
  `BottomUpAgent/group_a/Eye.py`
  `BottomUpAgent/group_a/Detector.py`
  `BottomUpAgent/group_a/Hand.py`
  `BottomUpAgent/group_a/UnifiedOperation.py`
  `BottomUpAgent/group_a/visualizer.py`

- **B组负责的代码主要放在**
  `BottomUpAgent/group_b/Brain.py`
  `BottomUpAgent/group_b/LongMemory.py`
  `BottomUpAgent/group_b/Mcts.py`
  `BottomUpAgent/group_b/Teacher.py`

- **公共协同代码主要放在**
  `run.py`
  `BottomUpAgent/common/BottomUpAgent.py`
  `base_model/`
  `config/`

#### 4. 这样拆分的意义

采用 `common/ + group_a/ + group_b/` 的目录组织方式，有两个明显好处：

1. **与小组分工完全一致** 
   文档中的职责划分可以直接映射到代码目录，老师或组员看到目录就能理解谁负责什么。

2. **更利于接口联调与后续维护** 
   A组 更关注“接入、执行、展示”，B组 更关注“识别、决策、记忆”，公共主控层负责把两者串联起来。这样既避免所有文件都堆在同一层，也避免把本来属于公共层的代码误划给某一组。

## 十、总结

整体来看，本项目并不是简单复现某个开源仓库，而是基于 Bottom-Up-Agent 的设计思想，对“自底向上智能体”这一前沿问题进行课程化理解、模块化拆解和工程化整理。

从系统结构上看，它围绕“感知—决策—执行—反馈—经验更新”构成了完整闭环；从分工上看，A组 负责交互执行与展示平台，B组 负责状态感知与策略学习引擎，公共模块则负责统一调度与运行支撑；从课程实践角度看，该项目能够较好体现人工智能课程中视觉感知、推理规划、动作控制、经验学习和系统集成等核心能力。

因此，本项目既具有一定的理论分析价值，也具有较强的工程展示价值，适合作为人工智能课程大作业的研究与汇报主题。
